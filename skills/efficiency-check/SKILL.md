# Efficiency Check — проверка эффективности

## Когда применять
**Всегда** при получении новой задачи или идеи от Дмитрия (или любого пользователя).

## Когда НЕ применять
- Задача помечена "просто сделай" / "без обсуждений"
- Мелкие правки (опечатки, однострочные изменения)
- Срочные задачи с явным дедлайном "прямо сейчас"

## Алгоритм (10 секунд)
1. **Есть ли готовое?** Проверь существующие скрипты, скиллы, инструменты в проекте
2. **Есть ли проще?** Можно ли достичь 80% результата за 20% усилий?
3. **Есть ли автоматизация?** pip/npm пакет, MCP-сервер, API, готовый скилл на skillsmp.com?

## Формат ответа
Если нашёл лучший путь — **1-2 предложения** + вопрос:

> 💡 [Альтернативный подход]. Делаем так или по-твоему?

Если лучшего пути нет — молча выполняй задачу.

## Для Telegram-ботов (HQ, Виктория)
Ещё короче — 1 предложение максимум:

> 💡 Можно проще: [как]. Делаем?

## Примеры

**Задача:** "Напиши скрипт парсинга Instagram"
> 💡 У нас уже есть `scripts/ig-parser.py` — можно добавить нужную команду туда. Делаем так или новый скрипт?

**Задача:** "Сделай презентацию в Google Slides"
> 💡 HTML-презентация будет быстрее и гибче для OBS. Плюс не нужен Google-аккаунт. Делаем так или через Slides?

**Задача:** "Напиши бота для напоминаний"
> [молчим — уже есть бот, но задача может быть про другой контекст, уточняем]

## Рычаги оптимизации (High-ROI)

### 1. Output Discipline
Maintenance-скрипты и cron-задачи должны быть тихими при успехе. Логируй ошибки, не успехи.
- ❌ "Heartbeat OK at 15:30" каждые 5 минут
- ✅ Тишина при успехе, алерт при сбое

### 2. Разделение работы и уведомлений
Длинные задачи выполняй тихо, отправляй короткий receipt по завершении.
- ❌ "Начинаю... Шаг 1... Шаг 2... Готово!"
- ✅ [тишина] → "✅ Задача X выполнена за 3 мин"

### 3. Bootstrap Discipline
Файлы, загружаемые в КАЖДЫЙ разговор (CLAUDE.md, MEMORY.md) должны быть короткими.
- CLAUDE.md: правила + таблица маршрутизации (компактно)
- MEMORY.md: индекс со ссылками, не контент
- Длинные инструкции → в references/ или отдельные файлы

### 4. Ambient Specialist Reduction
Не держи все скиллы "включёнными". Загружай по запросу.
- ❌ 20 скиллов в системном промпте
- ✅ Таблица маршрутизации → загрузка по триггеру

### 5. Verification-First Ops
После любой оптимизации проверяй:
- [ ] Основная функция работает
- [ ] Поведение не изменилось
- [ ] Контекст не раздулся
- [ ] Есть путь отката

### 6. Smallest Viable Change
Начинай с минимального изменения. Не оптимизируй всё сразу.
- Одно изменение → проверка → следующее
- Откатывай если improvement < 10%
