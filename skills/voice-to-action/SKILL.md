# voice-to-action

## Purpose

Обработка голосовых сообщений из Telegram: извлечение интентов, классификация, маршрутизация к соответствующим скиллам и выполнение после подтверждения. Превращает неструктурированную речь в конкретные действия, при этом всегда запрашивая подтверждение перед исполнением. Скилл работает как диспетчер между голосовым вводом и всей экосистемой агента.

## When to Use

- Поступило голосовое сообщение (Telegram voice → Whisper/TG Premium → текст)
- Пользователь отправил несколько голосовых подряд (batch-режим)
- Текст помечен как транскрипция голосового сообщения
- Запрос содержит маркеры: «из голосового», «голосовое», «voice message»
- Агент получил текст с характерными ASR-артефактами (нет пунктуации, оговорки, повторы)

## Core Workflow

### Phase 1: Receive — приём транскрипции

1. Получить текст транскрипции из voice handler бота.
2. Зафиксировать метаданные: timestamp, user_id, длительность аудио (если доступно).
3. Очистить текст от явных ASR-мусорных токенов (повторы слов, «ээ», «ммм»).
4. Если поступило несколько голосовых — объединить в единый контекст с разделителями `[Голосовое N]`.

### Phase 2: Classify — классификация интентов

1. Разбить текст на смысловые блоки (одно голосовое может содержать 1-5 интентов).
2. Для каждого блока определить тип интента по таблице классификации (см. ниже).
3. Извлечь параметры: кто, что, когда, куда, сколько.
4. Если интент неоднозначен — пометить как `unclear` и подготовить уточняющий вопрос.
5. Присвоить каждому интенту уровень уверенности: high / medium / low.

### Phase 3: Present — представление результатов

1. Показать {{OWNER_NAME}} пронумерованный список извлечённых интентов.
2. Формат представления:

```
Из голосового выделил N действий:

1. [task] Обновить каталог → product-catalog
2. [booking] Записать Машу на МК в субботу → booking-scheduler
3. [idea] МК для детей → IDEAS_AND_FEATURES.md
4. [unclear] "...что-то про Оксану завтра..." → уточнить

Подтвердить все / выбрать номера / отклонить?
```

3. Для каждого интента с уверенностью `low` — явно показать исходный фрагмент текста.
4. Для `unclear` — задать конкретный вопрос: «Что именно сделать с Оксаной завтра: написать, позвонить, встретиться?»

### Phase 4: Confirm — подтверждение

<HARD-GATE>
НЕ выполнять НИ ОДНОГО действия без явного подтверждения от {{OWNER_NAME}}.
Это правило НЕ имеет исключений. Даже если интент кажется очевидным —
запросить подтверждение. Confirmation gate is NON-NEGOTIABLE.
</HARD-GATE>

1. Принять ответ: «да» / «все» / номера через запятую / «нет» / «отмена».
2. «Да, кроме 3» — выполнить все кроме указанных.
3. При частичном подтверждении — отложить неподтверждённые (не удалять).
4. При полном отклонении — спросить, что было не так (данные для обучения).

### Phase 5: Route — маршрутизация и выполнение

1. Для каждого подтверждённого интента вызвать соответствующий скилл.
2. Передать извлечённые параметры в формате, ожидаемом целевым скиллом.
3. При ошибке выполнения — сообщить {{OWNER_NAME}}, не пытаться исправить молча.
4. Собрать результаты всех выполненных действий в итоговый отчёт.

### Phase 6: Log — логирование

1. Записать все интенты (подтверждённые и отклонённые) в лог.
2. Зафиксировать: исходный текст, извлечённый интент, уверенность, результат (confirmed/rejected/modified), целевой скилл, статус выполнения.
3. Зафиксировать паттерны в diary бота для self-learning (тип интента, confirmed/rejected).

## Intent Classification Table

| Intent | Trigger-слова (по смыслу) | Параметры | Route to |
|--------|--------------------------|-----------|----------|
| task | нужно, сделай, обнови, исправь, подготовь, настрой | что, когда, приоритет | Определяется по контексту задачи |
| booking | запиши, записать, забронируй, хочу на МК, консультация | кого, куда, дата, время | booking-scheduler |
| post | запости, опубликуй, выложи, пост, сторис | что, куда, когда | cross-poster |
| idea | а что если, было бы круто, идея, может сделать | описание идеи | IDEAS_AND_FEATURES.md |
| question | сколько, когда, кто, есть ли, покажи, статус | предмет вопроса | Определяется по контексту вопроса |
| reminder | напомни, не забыть, завтра утром, через час | что, когда, кому | cron / calendar (icloud_calendar.py) |
| decision | да отправляй, подтверждаю, согласен, давай так | ссылка на pending action | confirm pending action |
| message | напиши [кому], скажи [кому], передай | кому, текст | dm-sender / tg-send.py |
| catalog | каталог, товар, цена, прайс, обнови каталог | действие, товар | product-catalog |

**Правило маршрутизации:** если интент попадает под несколько категорий — выбрать наиболее специфичную. «Запиши Машу на МК» = booking (не task). «Обнови прайс» = catalog (не task).

## Proactive Behaviors

### Паттерн 1: Сводка после обработки
После выполнения всех действий показать краткий итог:
```
Выполнено 3 из 4 действий:
- Каталог обновлён
- Маша записана на субботу 14:00
- Пост запланирован на 18:00
- [отклонено] МК для детей — не подтверждено
```

### Паттерн 2: Обнаружение повторяющихся голосовых запросов
Если один и тот же тип запроса поступает голосом 3+ раз за неделю:
```
Заметил: вы часто голосом просите [обновить каталог].
Создать быструю команду /update_catalog? (Да / Нет)
```

### Паттерн 3: Batch-обработка серии голосовых
Если поступило 2+ голосовых подряд — объединить все интенты в единый пронумерованный список. Не показывать каждое голосовое отдельно. Группировать по типу интента.

### Паттерн 4: Контекстное обогащение
Если в голосовом упомянуто имя клиента — подтянуть контекст из досье (`employees/`, если существует; иначе использовать diary бота). Если упомянута дата «завтра» — подставить конкретную дату. Если «как обычно» — найти последний аналогичный запрос в diary.

## Self-Learning (Feedback Loops)

### Loop 1: Подтверждение vs отклонение
- Отслеживать процент подтверждённых интентов по каждому типу.
- Если rejection rate для типа > 30% за неделю — пересмотреть критерии классификации.
- Логировать причину отклонения (если {{OWNER_NAME}} указал).

### Loop 2: Паттерны голосовых команд
- Собирать частотные n-граммы из подтверждённых голосовых.
- Формировать словарь «голосовых shortcut-ов» {{OWNER_NAME}}.
- Предлагать создание быстрых команд для TOP-3 паттернов.

### Loop 3: FAQ из голосовых
- Если один и тот же вопрос задаётся голосом 3+ раз — предложить добавить в FAQ бота.
- Вопросы клиентов (через ботов пользователей) — отдельная категория: фиксировать в knowledge/.

### Loop 4: Точность ASR
- Если {{OWNER_NAME}} исправляет распознанный текст — логировать пару (ошибка → правильно).
- Собирать словарь частых ASR-ошибок для конкретного пользователя.

## Voice Error Handling (ASR Artifacts)

### Принцип: Match by MEANING, not by exact words

Транскрипция голосовых содержит ошибки. Классификация интентов ВСЕГДА работает по смыслу, а не по точному совпадению слов.

### Типичные ASR-ошибки и стратегии

| Ошибка | Пример | Стратегия |
|--------|--------|-----------|
| Пропуск пунктуации | "запиши машу на мк в субботу и ещё напомни позвонить" | Разбить по смысловым союзам (и, ещё, а также, потом) |
| Слитные слова | "мастеркласс", "наканалв" | Approximate similarity matching: Claude оценивает визуальное и фонетическое сходство слов |
| Омофоны | "код" / "кот", "мак" / "МК" | Выбирать по контексту бизнеса |
| Имена собственные | "Машу" / "машу", "Оксане" / "Аксане" | Сопоставлять с известными контактами из employees/ |
| Числа словами | "в три часа", "пятнадцатого" | Конвертировать в числовой формат |
| Повторы и заполнители | "ну вот эээ значит нужно это вот" | Удалить noise-токены перед классификацией |
| Незавершённые фразы | "и ещё хотел сказать что..." | Пометить как incomplete, запросить уточнение |

### Словарь бизнес-терминов

Для повышения точности распознавания поддерживать словарь бизнес-специфичных терминов клиента (названия товаров, имена сотрудников, услуги). Хранить в `knowledge/voice-dictionary.json`.

## Anti-Patterns

1. **НЕ выполнять действия без подтверждения.** Даже если интент очевиден на 100% — confirmation gate обязателен. Единственное исключение: intent type = `decision` (пользователь явно подтверждает ранее предложенное действие).

2. **НЕ угадывать при неясном интенте.** Если уверенность ниже medium — задать уточняющий вопрос. Лучше спросить один раз, чем откатывать неправильное действие.

3. **НЕ игнорировать часть интентов.** Если в голосовом 5 интентов — показать все 5. Не «оптимизировать» пропуская мелкие.

4. **НЕ отправлять сообщения людям, не упомянутым в голосовом.** Маршрутизация сообщений строго по именам из транскрипции. Не додумывать получателей.

5. **НЕ обрабатывать голосовое как обычный текст.** Всегда проходить через полный пайплайн: clean → classify → present → confirm → route → log. Пропуск любого этапа запрещён.

6. **НЕ хардкодить имена пользователей, клиентов, скиллов.** Использовать плейсхолдеры и динамический lookup по таблице маршрутизации.

7. **НЕ выполнять batch-действия параллельно без показа плана.** Сначала показать весь список, потом выполнять. Даже если 10 интентов — сначала план.

8. **НЕ удалять отклонённые интенты из лога.** Отклонения — ценные данные для self-learning. Логировать с пометкой `rejected`.

## Tools & Scripts

Этот скилл НЕ требует отдельных скриптов. Claude выполняет классификацию, извлечение параметров и маршрутизацию inline. Транскрипция обеспечивается встроенным пайплайном бота.

| Tool | Назначение | Статус |
|------|-----------|--------|
| Транскрипция | Deepgram nova-2 (primary) + faster-whisper (fallback) | Встроено в voice handler бота (`bot/engine/transcribe.py`) |
| tg-send.py | Отправка сообщений при intent=message | `python3 scripts/tg-send.py <user> "text"` |
| icloud_calendar.py | Создание напоминаний при intent=reminder | `python3 scripts/icloud_calendar.py add --title "X" --date Y --time Z` |
| skill-tracker.py | Учёт использования скилла | `python3 scripts/skill-tracker.py use voice-to-action` |
| vsearch | Семантический поиск (для контекста) | `vsearch "запрос"` |

**Логирование:** Intent-логи ведёт Claude в формате diary (встроен в бота). Для продвинутой аналитики — в будущем создать `logs/voice-intents/` (Phase 3).

## Portability

Скилл не привязан к конкретной капсуле. Для подключения к новому боту:
1. Убедиться что voice handler транскрибирует аудио в текст.
2. Передать транскрипцию в агента с пометкой `[voice]`.
3. Настроить таблицу маршрутизации под скиллы конкретной капсулы.
4. Создать `knowledge/voice-dictionary.json` с терминами бизнеса клиента.
